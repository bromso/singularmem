# TypeScript SDK Foundation + Reads (Sub-project 5a) — Design Spec

**Date:** 2026-05-18
**Status:** Approved (pending user review of written spec)
**Sub-project:** 5a (first of three TS SDK sub-projects)

## Summary

Introduce a new workspace crate `singularmem-node` that uses napi-rs to expose
the read-side of `singularmem-core` to Node.js / TypeScript consumers as a
native addon. This is the foundation sub-project: it establishes the crate
layout, npm package, build pipeline, type bridge, error mapping, and three-layer
test strategy. Search/retrieve bindings come in 5b; writes come in 5c;
multi-platform packaging comes in sub-project 6.

JS-based agents and applications will be able to open a Singularmem store and
perform read operations (`get`, `list`, `revisions`, `formatVersion`, `export`)
in-process, without spawning the CLI or the MCP server.

## Motivation

Singularmem currently has two consumer surfaces: the `singularmem` CLI (Rust
process) and the `singularmem-mcp` MCP server (Rust process, JSON-RPC over
stdio). Both require spawning a child process. JS-based AI agent frameworks
(LangChain.js, Vercel AI SDK, Mastra, custom Node tooling) would benefit from a
native in-process binding — lower latency, no child process management, native
async/await ergonomics, and full TypeScript types.

napi-rs is the de-facto standard for Rust → Node bindings. It's used by
Next.js (SWC), Rspack, lightningcss, oxc, and many others. It produces a
single `.node` cdylib per platform and generates `index.js` + `index.d.ts`
automatically from `#[napi]` annotations.

**Crate:** registered in root `Cargo.toml` `[workspace] members`. Version

## Section 2 — TypeScript API surface

```typescript
export interface StoreOptions {
  readOnly?: boolean;
}

export interface ListOptions {
  tags?: string[];
  limit?: number;
}

export interface Item {
  id: string;
  content: string;
  createdAt: Date;
  supersedes?: string;
  tags: string[];
  source?: string;
  metadata: Record<string, unknown>;
}

export class Store {
  static open(path: string, options?: StoreOptions): Promise<Store>;
  get(id: string): Promise<Item>;
  list(options?: ListOptions): Promise<Item[]>;
  revisions(id: string): Promise<Item[]>;
  formatVersion(): Promise<string>;
  export(): Promise<string>;
}
```

**Decisions locked in this section:**

- All methods are async (return `Promise<T>`); the underlying SQLite work runs
  via `tokio::task::spawn_blocking` to keep the JS event loop unblocked.
- `Store::open` is a static async factory (not a constructor) so it can be
  async and so errors during open are catchable with `try/await`.
- `list()` returns `Item[]` (not an async iterator). YAGNI — paginated/streaming
  reads can be added later without breaking the array-returning shape.
- `createdAt: Date` (not `number` or `string`). Accepts ms precision loss;
  documented in JSDoc.
- Errors are plain JS `Error` instances with a structured `code` field
  (see Section 3). No custom error subclasses.
- No `Store.ingest()`, no `Store.search()`, no `Store.retrieve()` in 5a — those
  are 5b/5c concerns.
- No hook registration in 5a (and likely indefinitely; see "Open Questions").

## Section 3 — Rust ↔ TypeScript type mappings

### Primitive + container mappings

| Rust | TypeScript | Notes |
|---|---|---|
| `String` | `string` | UTF-8 |
| `Vec<T>` | `T[]` | JS array |
| `Option<T>` | `T \| undefined` | `None` → `undefined`, never `null` |
| `bool` | `boolean` | — |
| `u32` / `i32` | `number` | Safe integer range |

### Identity + time

| Rust | TypeScript | Conversion |
|---|---|---|
| `ItemId` (ULID) | `string` | 26-char Crockford base32. Validated by `ItemId::from_str` on input; serialized via `Display` on output. Invalid input → `code: "InvalidId"` |
| `jiff::Timestamp` | `Date` | `ts.as_millisecond() as f64` → `new Date(ms)`. **Lossy:** Jiff has ns precision, JS Date has ms. Documented in JSDoc on `Item.createdAt` |

### Domain types

`singularmem_core::Item` maps 1:1 to the TS `Item` interface above.
`metadata` is `serde_json::Value` on the Rust side, exposed as
`Record<string, unknown>` via napi's `serde-json` feature. Callers narrow
types themselves.

### Error mapping

| Rust variant | JS `error.code` | JS `error.message` |
|---|---|---|
| `Error::NotFound { id }` | `"NotFound"` | `"item not found: <id>"` |
| `Error::Validation(msg)` | `"Validation"` | the validation message |
| `Error::SupersedesNotFound { id }` | `"SupersedesNotFound"` | `"supersedes target not found: <id>"` |
| `Error::Io(e)` | `"Io"` | underlying io error |
| `Error::Sqlite(e)` | `"Sqlite"` | underlying rusqlite error |
| napi-only: invalid ULID | `"InvalidId"` | `"invalid item id: <input>"` |
| napi-only: invalid path | `"InvalidStorePath"` | `"store path is not valid: <path>"` |

Implementation: `singularmem-node/src/error.rs` defines `From<singularmem_core::Error>`
for `napi::Error`, mapping each variant via `napi::Error::new(GenericFailure, msg).with_reason(code)`.
napi-rs attaches `reason` as the `.code` property on the JS error.

### napi-rs annotation patterns

- `#[napi]` on a struct → JS class
- `#[napi]` on an `impl` block → each method becomes a JS method
- `#[napi(factory)]` on `Store::open` → async static factory
- `#[napi(object)]` on `StoreOptions`, `ListOptions`, `Item` → plain JS objects
- `async fn` in Rust → JS `Promise` automatically

The crate is a hybrid Cargo crate + npm package. `npm` is the entry point so
JS consumers never invoke cargo directly. `@napi-rs/cli` (npm dev dependency)

## Section 5 — Testing strategy

Three layers, each catching a different class of bug.

### Layer 1: Rust unit tests (`src/*.rs`)

Standard `#[cfg(test)] mod tests` blocks. Test the Rust-side glue *before* it
crosses the napi boundary: type conversions, error mapping, ULID parsing.
Run via `cargo test -p singularmem-node`. No Node required.

### Layer 2: Node integration tests (`test/*.test.mjs`)

End-to-end via `node --test` + `node:assert/strict`.

- `store_open.test.mjs` — open at tempfile, open with `{ readOnly: true }`,
  open invalid path → `code: "InvalidStorePath"`
- `store_get.test.mjs` — seeded store, `get(id)` returns `Item` with correct
  field types; `get(bogusId)` → `code: "NotFound"`; `get("not-a-ulid")` → `code: "InvalidId"`
- `store_list.test.mjs` — 3 items, asserts count, tag filtering, limit
- `store_revisions.test.mjs` — seeded supersession chain, asserts ordering
- `store_meta.test.mjs` — `formatVersion()` semver string; `export()` JSONL
  with meta line first

**Seeding pattern:** integration tests use the root `singularmem` CLI to
populate a tempdir store before calling into the bindings. This proves
cross-language file-format compatibility for free.

### Layer 3: Type-level smoke test (`test/types.test.mts`)

Compilation test that catches `.d.ts` drift. Imports the package, assigns
to typed variables, asserts compiles via `tsc --noEmit`. Adds `typescript`
as a devDep — small price for type guarantees.

### CI

One new blocking job in `.github/workflows/ci.yml`:

```yaml
node-bindings:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with: { node-version: '20' }
    - uses: dtolnay/rust-toolchain@stable
    - run: cd crates/singularmem-node && npm install && npm run build
    - run: cd crates/singularmem-node && npm test
    - run: cd crates/singularmem-node && npx tsc --noEmit test/types.test.mts
```

Linux only in 5a. Cross-platform CI is sub-project 6.

## Section 6 — Acceptance criteria + Constitution Check

### Acceptance criteria

**Crate + package:**
- `crates/singularmem-node/` exists matching the Section 1 layout
- `Cargo.toml` declares `crate-type = ["cdylib"]` and is in `[workspace] members`
- `package.json` names the package `singularmem`, version inherits from workspace
- `npm install && npm run build` from clean checkout produces a working `.node`

**API surface (matches Section 2 exactly):**
- `Store.open(path, options?): Promise<Store>` — async static factory
- `store.get(id): Promise<Item>` — `code: "NotFound"` for missing IDs
- `store.list(options?): Promise<Item[]>` — supports `{ tags, limit }`; returns items ordered oldest → newest by `created_at` (matches `singularmem-core::Store::list` / `list_by_tags`, which both `ORDER BY created_at ASC`). `limit` is applied in the binding layer by truncating the collected iterator (core's `list_by_tags` does not take a limit param)
- `store.revisions(id): Promise<Item[]>` — ordered oldest → newest
- `store.formatVersion(): Promise<string>` — semver
- `store.export(): Promise<string>` — JSONL with meta line first
- `Item`, `StoreOptions`, `ListOptions` interfaces match Section 2 verbatim
- All thrown errors have structured `code` field per Section 3 table

**Type mappings (per Section 3):**
- `ItemId` → `string`; invalid → `code: "InvalidId"`
- `jiff::Timestamp` → `Date` (ms precision, JSDoc-documented)
- `Option<T>` → `T | undefined` (never `null`)
- `serde_json::Value` metadata → `Record<string, unknown>`

**Tests (per Section 5):**
- `cargo test -p singularmem-node` passes
- `cd crates/singularmem-node && npm test` passes
- `npx tsc --noEmit test/types.test.mts` passes
- All three layers run in CI as a blocking job

**Read-only enforcement:**
- `Store.open(path, { readOnly: true })` opens via core's
  `StoreOptions { read_only: true }`, matching the defense pattern from
  singularmem-mcp 4b

### Out of scope

- `ingest()` (5c)
- Search / retrieve bindings (5b)
- Prebuilt binaries, npm publish, cross-platform builds (sub-project 6)
- Hook registration
- GUI work (sub-project 7+)

### Constitution Check (v0.2.0)

- **I. Local-first, file-based** ✅ — same SQLite file, no network
- **II. Provider-agnostic** ✅ — adapters not touched in 5a
- **III. Append-only, revisable** ✅ — 5a is read-only; semantics unchanged
- **IV. Open core** ✅ — Apache-2.0, lives in `crates/`
- **V. Stable on-disk format** ✅ — `formatVersion()` newly exposed
- **VI. Single binary, zero deps** — partial / acceptable trade-off for language
  bindings; consumer needs Node 20+. Rust side unchanged.
- **VII. Composable crates** ✅ — depends only on `singularmem-core`
- **VIII. Tested at every layer** ✅ — three test layers, all blocking in CI
- **IX. Documented behavior** ✅ — JSDoc on every method, `.d.ts` committed, README
- **X. Performance budgets** — N/A for 5a; napi overhead negligible vs SQLite I/O.
  Hybrid/retrieve budgets come into play when 5b exposes those paths.

## Open questions (deferred, not blocking)

- Async iterator for `list()` — revisit in 5b/5c if real demand emerges
- Hook bindings — `&dyn` traits don't bridge cleanly through napi;
  deferred indefinitely
- Binary content via TypedArray — not in scope; content stays `string` (UTF-8)
