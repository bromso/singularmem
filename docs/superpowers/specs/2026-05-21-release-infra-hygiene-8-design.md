# Release-infra hygiene (Sub-project 8) — Design Spec

**Date:** 2026-05-21
**Status:** Draft (awaiting user review of written spec)
**Sub-project:** 8 (Release-infra hygiene)
**Builds on:** 6 (`v0.14.0`, npm distribution) and 7 (`v0.15.0`, cargo-dist CLI binary distribution).

## Summary

Pays down the technical debt accumulated while shipping sub-project 7's first cargo-dist release. Two coupled changes ship in a single PR:

1. **GitHub Actions bump** across all workflows from versions running on Node 20 to versions running on Node 24 (Node 20 actions hit a hard deprecation cutoff on 2026-06-02 — 12 days from the spec date).
2. **`release.yml` regeneration** via `dist init` against cargo-dist 0.31.0 templates, removing the five manual edits made during v0.15.0's release saga and the `allow-dirty = ["ci"]` escape-hatch in `dist-workspace.toml`.

No library code changes. Surface is `.github/workflows/*.yml`, `Cargo.toml`, `dist-workspace.toml`. The follow-on tag `v0.16.0` exercises the cleaned-up pipeline end-to-end.

## Problem & motivation

Sub-project 7 shipped the first cargo-dist release (`v0.15.0`), but landing it took five commits and six tag-pushes because cargo-dist 0.27.0 emitted GitHub-hosted runner labels that GitHub had retired (`ubuntu-20.04` Apr 2025, `windows-2019` Jun 2025, `macos-13` Oct 2025). The fix path bumped cargo-dist to 0.31.0, added `allow-dirty = ["ci"]` to silence the consistency check that would otherwise reject manual workflow edits, hardcoded the new installer URL, and dropped Windows MSVC from the platform matrix due to an ort-sys CRT mismatch.

The result is a working release pipeline that ships from `main` cleanly — but `release.yml` now contains hand-applied patches that drift from cargo-dist's canonical templates. Each future cargo-dist bump risks merge-conflict-style breakage. `allow-dirty` is an escape-hatch, not a destination.

Separately, every workflow currently emits "Node.js 20 actions are deprecated" warnings. After 2026-06-02, GitHub forces Node 24 on the runner by default; after 2026-09-16, the Node 20 runtime is removed entirely. Without action, every workflow breaks on June 2.

These two issues share a common surface (`.github/workflows/*`) and a common timing constraint (the regeneration of `release.yml` will incidentally pick up newer action versions, so bumping actions in other workflows in lockstep avoids duplicate work). Bundling them into one focused PR pays both debts at once.

## Goals & non-goals

### Goals

1. All GitHub Actions in `.github/workflows/*.yml` reference versions whose underlying JS runtime is Node 24 (or higher) — verified by absence of the "Node.js 20 actions are deprecated" annotation on any workflow run.
2. `release.yml` is regenerated by `dist init` against cargo-dist 0.31.0 templates with zero post-generation manual edits in the final state, OR with documented residual edits captured in this spec.
3. `allow-dirty = ["ci"]` is removed from `dist-workspace.toml`.
4. The release pipeline shipped by this sub-project (tag `v0.16.0`) produces:
   - A complete GitHub Release with 3-platform archives (linux x64, darwin x64, darwin arm64) for both `singularmem` and `singularmem-mcp` binaries + installers + checksums.
   - Updated Homebrew formulas in `bromso/homebrew-tap` at version `0.16.0`.
   - End-to-end install via `brew install bromso/tap/singularmem` and the curl-bash installer.

### Non-goals

- **Windows MSVC binary fix.** The `ort-sys` / `cxx` CRT runtime-library mismatch on `x86_64-pc-windows-msvc` is real engineering work (likely requires switching to ort's `load-dynamic` feature or forcing CRT linkage via RUSTFLAGS). Tracked as a separate future sub-project.
- **Linux arm64 binary.** Still blocked on the fastembed → ureq → openssl-sys cross-compile path from sub-project 6. Separate sub-project candidate.
- **npm CI publish 404 (sub-project 6b candidate).** Investigation-heavy; may not be solvable without npm support. Manual publish remains the workaround. Tracked separately.
- **crates.io publish (`cargo install singularmem`).** Distribution-channel expansion; out of scope for a hygiene pass.
- **Library code changes.** Nothing in `crates/*/src/**` should change.
- **Performance budget revisions.** This sub-project doesn't touch any code path that perf budgets cover.

## Recommended approach

Single PR titled `chore(ci): sub-project 8 — release infra hygiene`. Five sequenced steps:

1. **Audit and bump GH Actions** in `ci.yml`, `dco.yml`, `npm-publish.yml` (any others touched after this draft). Bump `actions/checkout`, `actions/upload-artifact`, `actions/download-artifact`, `actions/setup-node`, `actions/cache` from `@v4` to whatever current major emits Node-24-compatible bytecode (likely `@v5`). Skip `rustsec/audit-check@v1` (separate lineage; revisit if it warns).

2. **Operational pre-req — manual npm publish of `singularmem@0.15.0`.** Runs against `crates/singularmem-node/RELEASING.md`'s manual flow with the maintainer's TOTP. Confirms v0.15.0 is fully released before stacking v0.16.0 changes. Not part of the PR diff.

3. **Install cargo-dist 0.31.0 locally** (`curl --proto '=https' --tlsv1.2 -LsSf https://github.com/axodotdev/cargo-dist/releases/download/v0.31.0/cargo-dist-installer.sh | sh`). Run `dist init`. Inspect the diff.

4. **Reconcile the regenerated workflow against preservation constraints** (see "Constraints" below). Revert any unintended changes explicitly in the same commit. Drop `allow-dirty = ["ci"]` from `dist-workspace.toml` if the final shape no longer requires it.

5. **Open the PR.** `pr-run-mode = "upload"` validates the build matrix on the PR run. Tag a throwaway `v0.16.0-rc1` against the PR branch (or against a temporary `main`-aligned commit) to dry-run the full host + publish-homebrew + announce DAG. After RC success, delete the rc tag + release, merge the PR, bump workspace version `0.15.0 → 0.16.0` in a follow-up commit on main per existing convention, tag `v0.16.0`.

### Approaches discarded

- **Approach B — Two sequential PRs.** Bump actions first, then run `dist init`. Rejected because the two changes share `release.yml` (which `dist init` rewrites wholesale) and serializing them adds re-tag churn without giving any isolation benefit; the deadline pressure on Node 20 makes the serial path slower.
- **Approach C — Two parallel PRs.** Same surface conflict as B plus rebase work; worst of both worlds for this case.

## Constraints

`dist init` is opinionated about workflow shape, profile flags, and platform targets. The following session-accumulated decisions must survive regeneration; the PR must explicitly revert any unintended reversions and document residual ones:

| Constraint | Where it lives | Why preserve |
|---|---|---|
| 3-platform targets (no `x86_64-pc-windows-msvc`) | `dist-workspace.toml` `targets = [...]` | ort-sys CRT mismatch on Windows; separate sub-project will re-add when fixed |
| `[profile.dist]` has no `lto = "thin"` | root `Cargo.toml` | LTO removal isn't load-bearing for Windows (CRT mismatch persisted without it), so the decision point at implementation time is whether to tolerate `dist init` re-adding it. **Preferred:** tolerate re-add (less drift); Linux/Darwin tolerate LTO fine |
| `publish-jobs = ["homebrew"]` | `dist-workspace.toml` | Required for Homebrew publish job to fire (default-off in cargo-dist 0.27+) |
| `pr-run-mode = "upload"` | `dist-workspace.toml` | Catches cross-platform breakage on PR runs |
| `tap = "bromso/homebrew-tap"` | `dist-workspace.toml` | Self-hosted tap repo |
| `install-path = "CARGO_HOME"` | `dist-workspace.toml` | Installer drops binaries in CARGO_HOME |
| Both `singularmem` (CLI) AND `singularmem-mcp` (MCP server) ship in every Release archive | implicit via workspace shape | Sub-project 7's deliverable |

**Verification step at implementation time:** after `dist init`, diff `Cargo.toml`, `dist-workspace.toml`, `release.yml` against the current `main` HEAD; for each non-trivial diff, classify as "accept regenerated" or "revert to current"; document each classification in the PR description.

## Architecture

No new components, no new library boundaries. Affected surfaces:

- **`.github/workflows/ci.yml`** — Rust build/test/clippy/audit + node-bindings + DCO jobs. Bump action versions.
- **`.github/workflows/dco.yml`** — DCO enforcement. Bump action versions.
- **`.github/workflows/npm-publish.yml`** — napi-rs cross-platform build matrix + (broken) npm publish job. Bump action versions. The 404-on-PUT publish bug is explicitly out of scope.
- **`.github/workflows/release.yml`** — regenerated wholesale by `dist init`. Subsequent verification of preservation constraints.
- **`Cargo.toml`** — possibly touched by `dist init` (`[profile.dist]` shape). Reconcile per constraints.
- **`dist-workspace.toml`** — `cargo-dist-version` already at 0.31.0 (set in sub-project 7 saga). Drop `allow-dirty = ["ci"]`. Re-confirm `publish-jobs`, `pr-run-mode`, `tap`, `targets`, `install-path`.

## Data model

None. No on-disk format changes, no schema migrations, no Tantivy/USearch sidecar layout changes. Library-touching surfaces are not in scope.

## Interfaces

No CLI, library, or wire-protocol changes. The CLI version reported by `singularmem --version` advances `0.15.0 → 0.16.0` as part of the version-bump commit; the binary surface is otherwise identical.

## Error handling

Operational, not code:

- **`dist init` produces unexpected diffs.** Reconciled per Constraints table; documented in PR description.
- **Bumped action major version has breaking changes.** Read each action's CHANGELOG between `@v4` and `@v5` before bumping. If a breaking change is in scope (e.g., `actions/upload-artifact@v5` defaults change), update workflow callsites in the same PR.
- **`v0.16.0-rc1` release dry-run fails at `host` or `publish-homebrew-formula`.** Delete the rc tag + release; iterate on the PR; re-tag a fresh rc. Do NOT tag `v0.16.0` proper until rc succeeds end-to-end.
- **Manual npm publish step 2 takes longer than the Node 20 deadline.** Split the Node 20 actions bump out as a standalone hotfix PR before this sub-project lands; absorb the `dist init` work into a later PR. Hotfix scope: ONLY the `@v4 → @v5` bumps in `ci.yml`, `dco.yml`, `npm-publish.yml`.

Per Principle VII, all failure modes preserve state — no destructive operations (no force-push to main, no tag overwrites of `v0.15.0`).

## Testing strategy

No new unit or integration tests; no library code changes. Verification is operational:

**On the PR (before merge):**
1. CI workflow runs green — Rust build, tests, fmt, clippy, perf-budgets, audit, DCO, node-bindings — all jobs run on bumped actions.
2. `release.yml` runs in `upload` mode (per `pr-run-mode = "upload"`) — 3-platform build matrix completes successfully. Artifacts downloadable from the run for spot-inspection.
3. No "Node.js 20 actions are deprecated" warnings in any workflow's annotations.

**On main after merge + version bump + `v0.16.0` tag:**
1. `release.yml` triggers; full DAG green: `plan → build × 3 → build-global → host → publish-homebrew-formula → announce`.
2. GH Release `v0.16.0` published with: 3 platform archives × 2 binaries + checksums, 2 shell installer scripts (`singularmem-installer.sh` + `singularmem-mcp-installer.sh` — cargo-dist omits the PowerShell installer when no Windows target is in the matrix, observed in v0.15.0), Homebrew formulas, dist-manifest, source tarball.
3. `bromso/homebrew-tap` updated to `0.16.0` for both `singularmem.rb` and `singularmem-mcp.rb`.
4. `brew install bromso/tap/singularmem` + `singularmem --version` → `0.16.0` on a clean macOS machine.
5. curl-installer + `singularmem --version` → `0.16.0` on a clean Linux machine.

Per Principle VI: existing test suites must continue to pass with networking disabled — not changed by this sub-project, but re-verified.
Per Principle III.b: no closed-source dependency introduced — sub-project is `.github/workflows/*` only.

## Open questions

1. **`actions/upload-artifact@v5`** (and siblings) — does the major version exist yet, and does it default to behaviors compatible with `download-artifact@v5`? Implementation must confirm before bumping; if `@v5` is not yet GA for a given action, fall back to setting `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true` at the workflow level as documented in GitHub's deprecation notice.
2. **`dist init` and `[profile.dist] lto = "thin"`** — does cargo-dist 0.31.0 still emit `lto = "thin"` by default? If yes, the implementation accepts the re-add (per Constraints). If `dist init` provides a config knob to suppress the lto emission, that's preferred.
3. **`v0.16.0-rc1` dry-run mechanics** — RC tags trigger `release.yml` like any other `v*.*.*` tag. They WILL run the Homebrew publish job. The implementation must temporarily disable the `publish-jobs = ["homebrew"]` for the RC run OR scope `HOMEBREW_TAP_TOKEN` revocation as the rollback path. Implementation can choose; preferred: keep the homebrew publish enabled for the RC, then `git revert` the formula PR in `bromso/homebrew-tap` if the rc is rejected, so the rc fully exercises the same DAG as the real release.

## Acceptance criteria

Numbered, observable, testable. The sub-project is done when all items are observable on `main` and at the tagged release:

1. **No Node 20 deprecation warnings.** `gh run view <run-id>` on the CI run for the merge commit shows no "Node.js 20 actions are deprecated" annotation on any workflow.
2. **`dist-workspace.toml` has no `allow-dirty` key.** Verified by `grep -q '^allow-dirty' dist-workspace.toml || echo absent` → `absent`.
3. **`release.yml` matches `dist init` output bit-for-bit, modulo residual edits explicitly listed in the PR description.** Verified by running `dist init` again against the merged main; if it produces no diff, OR only diffs matching the residual-edits list in the PR description, pass. The PR description's residual-edits list is the single source of truth for "documented".
4. **PR CI green.** All jobs in `ci.yml`, `dco.yml`, `npm-publish.yml`, `release.yml` (in upload mode) pass on the PR run.
5. **`v0.16.0` release pipeline succeeds end-to-end.** All 7 release-pipeline jobs (`plan`, `build × 3`, `build-global`, `host`, `publish-homebrew-formula`, `announce`) report `conclusion=success`.
6. **GH Release `v0.16.0` exists** with the expected 20 assets (matching v0.15.0's shape exactly): 3 platforms × 2 binaries × 2 files (archive + sha256) = 12, plus 2 shell installer scripts (`singularmem-installer.sh` + `singularmem-mcp-installer.sh`; no PowerShell installer since Windows isn't in the matrix), plus 2 Homebrew formulas (`singularmem.rb` + `singularmem-mcp.rb`), plus `source.tar.gz` + `source.tar.gz.sha256`, plus `dist-manifest.json` + `sha256.sum`.
7. **Homebrew tap updated.** `gh api repos/bromso/homebrew-tap/contents/Formula/singularmem.rb --jq '.content' | base64 -d | grep 'version "0.16.0"'` returns a match.
8. **`brew install bromso/tap/singularmem` + `singularmem --version` → `0.16.0`** on a clean macOS test environment.
9. **`curl ... singularmem-installer.sh | sh` + `singularmem --version` → `0.16.0`** on a clean Linux test environment.

## Constitution Check

| Principle | How this design complies |
|---|---|
| **I — Local-First and Sovereign** | No new network calls in library code; binaries continue to work fully offline post-install. |
| **II — Provider-Agnostic by Contract** | No adapter changes. All four constitutional providers (plain, claude, openai, gemini) continue to ship in the same binaries. |
| **III — Open Core with a Stable Boundary** | Sub-project surface is `.github/workflows/*` + `dist-workspace.toml` + `Cargo.toml` — all open-side. No closed-source coupling introduced. |
| **V — Composable Library Architecture** | No library API changes. |
| **VI — Deterministic and Offline-Testable** | No new tests; existing offline-testable suite is unchanged. |
| **X — Performance Budgets, Enforced in CI** | No performance-touching code changes. `[profile.dist]` shape changes (possible re-add of `lto = "thin"`) don't change CI's perf-budget profile (which uses `release`). |

Principles IV (CLI-First), VII (Honest Failure Modes), VIII (Privacy Telemetry), and IX (Accessible by Default) are not touched by this sub-project's surface.
