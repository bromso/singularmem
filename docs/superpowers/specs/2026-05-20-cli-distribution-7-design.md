# CLI + MCP Binary Distribution (Sub-project 7) — Design Spec

**Date:** 2026-05-20
**Status:** Approved (pending user review of written spec)
**Sub-project:** 7 (CLI + MCP binary distribution via cargo-dist)
**Builds on:** 6 (npm distribution for the TS SDK, `v0.14.0`). Independent of the sub-project 6b candidate (fixing CI npm publish 404).

## Summary

Ships prebuilt `singularmem` (CLI) and `singularmem-mcp` (MCP server) binaries via GitHub Releases, with three install paths for consumers: a curl/PowerShell installer script, a Homebrew tap (`bromso/homebrew-tap`), and manual download from the Releases page. Uses [`cargo-dist`](https://opensource.axo.dev/cargo-dist/) (latest pinned version: `0.27.0`) to generate the release workflow + per-platform archives + installers + Homebrew formula in lockstep with the existing `npm-publish.yml` workflow from sub-project 6.

Both binaries ship together in every Release (one archive per platform, contains both binaries). Same `v*.*.*` tag triggers both this `release.yml` workflow AND the existing `npm-publish.yml`. After this sub-project lands, **Singularmem's CLI is installable in one command on macOS, Linux x86_64, and Windows x86_64** — no Rust toolchain required.

## Motivation

Sub-projects 5a/5b/5c shipped the TS SDK (JavaScript / TypeScript surface). Sub-project 6 published it to npm. But the `singularmem` CLI itself (and the MCP server) still require `cargo install --git ...` or building from source. For non-Rust users — which is the majority of LLM tooling consumers — that's a non-starter.

Sub-project 7 closes the last major distribution gap before v1: anyone can install both binaries in one command. cargo-dist is the industry-standard tool for this (uv, ruff, pixi, biome, oxc all use it); adopting it gives us a battle-tested release pipeline + installer scripts + Homebrew integration without writing the underlying scaffolding ourselves.

CLI distribution via crates.io (`cargo install singularmem`), Linux package managers (apt/dnf/snap), and Windows/macOS native installers (MSI/pkg) are deferred — the three install paths in this sub-project cover ~95% of the target audience.

## Section 1 — `cargo-dist` overview + how it slots in

`cargo-dist` (binary name: `dist`) is a Rust-aware release-engineering tool. Once configured, the workflow is:

1. Maintainer pushes a `v*.*.*` tag
2. Existing `npm-publish.yml` triggers (sub-project 6 — publishes JS SDK packages)
3. NEW `release.yml` (auto-generated by `dist init`) triggers in parallel:
   - Builds the `singularmem` CLI + `singularmem-mcp` binaries on a per-platform matrix
   - Packages each platform as `singularmem-<version>-<triple>.tar.gz` (or `.zip` on Windows)
   - Creates a GitHub Release with all the archives + auto-generated installer scripts + SHA256 checksums
   - Pushes an updated formula to the separate `bromso/homebrew-tap` repo

Consumers get three install paths:

```bash
# 1. Manual: download from https://github.com/bromso/singularmem/releases/latest
#    + extract + put on PATH

# 2. Curl-bash installer (Linux + macOS)
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/bromso/singularmem/releases/latest/download/singularmem-installer.sh | sh

# 2b. PowerShell installer (Windows)
powershell -ExecutionPolicy ByPass -c "irm https://github.com/bromso/singularmem/releases/latest/download/singularmem-installer.ps1 | iex"

# 3. Homebrew tap (macOS + Linux)
brew install bromso/tap/singularmem
```

Each install path puts BOTH `singularmem` and `singularmem-mcp` binaries on the user's PATH.

**Decisions locked in this section:**
- Same `v*.*.*` tag triggers both `npm-publish.yml` AND the new `release.yml`. Workspace version is the single source of truth.
- Both binaries (`singularmem` + `singularmem-mcp`) bundle in every Release archive.
- 4 platforms: `linux x64-gnu`, `darwin x64`, `darwin arm64`, `win32 x64-msvc` (matches sub-project 6).
- Homebrew formula lives in a separate `bromso/homebrew-tap` repo (Homebrew convention for self-hosted taps).
- No binary provenance / sigstore signing in v0.

## Section 2 — Cargo.toml metadata + workspace config for `dist`

### Root `Cargo.toml` additions

```toml
[workspace.metadata.dist]
cargo-dist-version = "0.27.0"
ci = "github"
installers = ["shell", "powershell", "homebrew"]
targets = [
  "x86_64-unknown-linux-gnu",
  "x86_64-apple-darwin",
  "aarch64-apple-darwin",
  "x86_64-pc-windows-msvc",
]
auto-includes = true
include = []  # default: publish all crates with binary targets
tap = "bromso/homebrew-tap"
publish-prereleases = false
github-release = "auto"
pr-run-mode = "plan"
allow-dirty = []
```

The version line (`cargo-dist-version = "0.27.0"`) pins the exact version of `dist` that CI uses. `dist init` regenerates the workflow when this is bumped.

### Required Cargo.toml field updates (root + binary crates)

`dist` requires `description`, `repository`, and `license` on every crate that produces a publishable artifact. Both binary-producing crates already inherit these via the workspace pattern. The implementer should verify with `cargo dist plan` (a dry-run mode) before generating the workflow.

### Per-binary metadata

`dist` picks up binaries via the existing `[[bin]]` entries in each crate:

- Root crate: `[[bin]] name = "singularmem" path = "src/main.rs"`
- `singularmem-mcp` crate: `[[bin]] name = "singularmem-mcp" path = "src/main.rs"` (verify)

`dist` automatically includes both. No additional config needed.

### One adjustment: `publish` field sanity-check

`dist` only releases crates that have `publish = true` (the default). The implementer should sanity-check that nothing in the workspace sets `publish = false`, which would silently get skipped.

**Decisions locked in this section:**
- Pin `cargo-dist-version` to `0.27.0`. Avoids auto-update churn.
- 3 installer types: `shell` (curl-bash for Linux + macOS), `powershell` (Windows), `homebrew` (separate tap repo). Skip `npm` (separate concern — sub-project 6 handles npm packaging for the JS bindings; different package).
- `tap = "bromso/homebrew-tap"` — a separate GitHub repo to be created (Section 5).
- `auto-includes = true` — each archive contains README.md + LICENSE alongside the binaries.
- No crates.io publish in v0 (deferred).

## Section 3 — The auto-generated `release.yml` workflow

`dist init` generates `.github/workflows/release.yml` from the workspace config. The implementer doesn't write this file by hand — they run `dist init` once and commit whatever it produces.

### Structural skeleton (approximate; actual `dist init` output is more detailed)

```yaml
name: Release

permissions:
  contents: write
  pull-requests: write

on:
  push:
    tags: ['**[0-9]+.[0-9]+.[0-9]+*']
  pull_request:
    paths:
      - '.github/workflows/release.yml'

jobs:
  plan:
    runs-on: ubuntu-latest
    outputs:
      val: ${{ steps.plan.outputs.manifest }}
      publishing: ${{ !github.event.pull_request }}
    steps:
      - uses: actions/checkout@v4
      - name: Install dist
        run: curl --proto '=https' --tlsv1.2 -LsSf https://github.com/axodotdev/cargo-dist/releases/download/v0.27.0/cargo-dist-installer.sh | sh
      - id: plan
        run: dist plan --tag=${{ github.ref_name }} --output-format=json

  build-local-artifacts:
    needs: plan
    strategy:
      fail-fast: false
      matrix:
        include:
          - target: x86_64-unknown-linux-gnu
            runner: ubuntu-latest
          - target: x86_64-apple-darwin
            runner: macos-14
          - target: aarch64-apple-darwin
            runner: macos-14
          - target: x86_64-pc-windows-msvc
            runner: windows-latest
    runs-on: ${{ matrix.runner }}
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.target }}
      - name: Install dist
        run: curl --proto '=https' --tlsv1.2 -LsSf https://github.com/axodotdev/cargo-dist/releases/download/v0.27.0/cargo-dist-installer.sh | sh
      - run: dist build --artifacts=local --target=${{ matrix.target }}
      - uses: actions/upload-artifact@v4
        with:
          name: artifacts-${{ matrix.target }}
          path: target/distrib/*

  build-global-artifacts:
    needs: [plan, build-local-artifacts]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install dist
        run: <same as above>
      - uses: actions/download-artifact@v4
        with: { path: target/distrib, pattern: artifacts-*, merge-multiple: true }
      - run: dist build --artifacts=global

  host:
    needs: [plan, build-local-artifacts, build-global-artifacts]
    runs-on: ubuntu-latest
    if: needs.plan.outputs.publishing == 'true'
    steps:
      - uses: actions/checkout@v4
      - name: Install dist
        run: <same as above>
      - env: { GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }} }
        run: dist host --steps=upload --steps=release

  homebrew-tap:
    needs: [plan, host]
    runs-on: ubuntu-latest
    if: needs.plan.outputs.publishing == 'true'
    env:
      GITHUB_USER: bromso
      GITHUB_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}
    steps:
      - uses: actions/checkout@v4
      - run: dist generate --mode=ci-tap-homebrew
```

### End-to-end flow for a tag push

1. **plan**: figures out which binaries to build, which platforms, which installers to generate
2. **build-local-artifacts** (4 parallel matrix jobs): each builds the binaries for one platform, produces a `.tar.gz` (or `.zip` on Windows), uploads as a workflow artifact
3. **build-global-artifacts**: downloads all 4 platform artifacts, generates installer scripts (sh + ps1) + Homebrew formula
4. **host**: creates the GitHub Release at the tag, uploads all artifacts (per-platform archives + installers + checksums + Homebrew formula `.rb`)
5. **homebrew-tap**: pushes the updated formula to `bromso/homebrew-tap` via the `HOMEBREW_TAP_TOKEN` PAT

### PR behavior

Every PR runs `plan` + `build-local-artifacts` (the matrix). This catches platform-specific compile breakage early. The `host` and `homebrew-tap` jobs are gated on `publishing == 'true'`, which is only true for tag pushes — PRs don't publish.

### Differences from sub-project 6's npm-publish.yml

- **Built via `dist build`**, not raw `cargo build` + napi codegen
- **No `cross-rs` or `--zig`**. dist 0.27 only supports native or vendored cross-compilation. Linux arm64 excluded for the same openssl-sys reason from sub-project 6.
- **macOS x64 from `macos-14`** — same pattern as sub-project 6.
- **No `--provenance`**. dist doesn't generate npm-style provenance for binary distribution.

### Required GitHub repo permissions for the workflow

- `contents: write` (default `GITHUB_TOKEN`) — creates the GitHub Release + uploads assets
- `HOMEBREW_TAP_TOKEN` repo secret — a PAT with `Contents: write` on `bromso/homebrew-tap` (Section 5)

**Decisions locked in this section:**
- All 4 platforms build in parallel (matches sub-project 6).
- `dist init` writes the file; we don't hand-craft it. If `dist init` produces something materially different from this sketch, the implementer adopts whatever `dist` generates.
- PR workflow runs build-only (no publish) — proves cross-platform compile per-PR.
- No binary provenance attestation in v0.

## Section 4 — Install paths from the consumer's perspective

### 1. Homebrew tap (macOS + Linux power users)

```bash
brew install bromso/tap/singularmem
```

Homebrew clones (or updates) `bromso/homebrew-tap`, reads `Formula/singularmem.rb`, downloads the right archive from the GH Release, extracts, symlinks `singularmem` + `singularmem-mcp` to `/usr/local/bin/` (or `/opt/homebrew/bin/` on Apple Silicon).

Upgrades: `brew upgrade singularmem`.

Auto-generated formula:

```ruby
class Singularmem < Formula
  desc "Local-first persistent memory layer for LLM workflows"
  homepage "https://github.com/bromso/singularmem"
  version "0.15.0"
  license "Apache-2.0"

  on_macos do
    if Hardware::CPU.intel?
      url "https://github.com/bromso/singularmem/releases/download/v0.15.0/singularmem-x86_64-apple-darwin.tar.gz"
      sha256 "..."
    end
    if Hardware::CPU.arm?
      url "https://github.com/bromso/singularmem/releases/download/v0.15.0/singularmem-aarch64-apple-darwin.tar.gz"
      sha256 "..."
    end
  end

  on_linux do
    if Hardware::CPU.intel?
      url "https://github.com/bromso/singularmem/releases/download/v0.15.0/singularmem-x86_64-unknown-linux-gnu.tar.gz"
      sha256 "..."
    end
  end

  def install
    bin.install "singularmem"
    bin.install "singularmem-mcp"
  end

  test do
    assert_match(/singularmem \d+\.\d+\.\d+/, shell_output("#{bin}/singularmem --version"))
  end
end
```

### 2. Curl-bash / PowerShell installer

```bash
# macOS + Linux
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/bromso/singularmem/releases/latest/download/singularmem-installer.sh | sh
```

```powershell
# Windows
powershell -ExecutionPolicy ByPass -c "irm https://github.com/bromso/singularmem/releases/latest/download/singularmem-installer.ps1 | iex"
```

Script detects the platform, downloads the right archive, extracts to `~/.cargo/bin/` by default (configurable via `--install-dir`), optionally appends to PATH in `.bashrc` / `.zshrc` / `$PROFILE`.

### 3. Manual download from GitHub Releases page

Users navigate to https://github.com/bromso/singularmem/releases/latest and pick the right archive:

- `singularmem-x86_64-unknown-linux-gnu.tar.gz` (~35-40 MB compressed)
- `singularmem-x86_64-apple-darwin.tar.gz`
- `singularmem-aarch64-apple-darwin.tar.gz`
- `singularmem-x86_64-pc-windows-msvc.zip`

Each contains `singularmem`, `singularmem-mcp`, `README.md`, `LICENSE`.

### Why a separate `homebrew-tap` repo

Homebrew's tap-discovery convention: `brew install <owner>/<tap>/<formula>` looks for `<owner>/homebrew-<tap>` on GitHub. So `bromso/tap/singularmem` resolves to `github.com/bromso/homebrew-tap` → `Formula/singularmem.rb`.

The tap repo is just a small repo with a single `Formula/` directory. `dist`'s `homebrew-tap` job pushes the formula via a PAT (`HOMEBREW_TAP_TOKEN`), so the main repo's default `GITHUB_TOKEN` (scoped to this repo) doesn't have access.

### Verification consumers can run

```bash
singularmem --version    # → "singularmem 0.15.0"
singularmem-mcp --version    # → "singularmem-mcp 0.15.0"
singularmem --help        # → CLI usage
```

### What's NOT covered for install paths in v0

- `cargo install singularmem` (requires crates.io publish — deferred)
- `apt install singularmem` / `dnf install` / `snap install` (deferred indefinitely)
- Windows MSI installer (deferred; PowerShell installer covers the Windows case)
- macOS pkg installer (deferred; Homebrew + curl-installer cover macOS)

## Section 5 — Repository setup + one-time manual steps

### A) Create the `bromso/homebrew-tap` repository

1. https://github.com/new
2. Repository name: **homebrew-tap** (exact spelling — Homebrew convention requires the `homebrew-` prefix on the GitHub repo)
3. Owner: `bromso`
4. Description: "Homebrew formulas for Singularmem and related tools"
5. License: Apache-2.0
6. **Public**
7. Initialize with a README:

   ```markdown
   # Singularmem Homebrew tap

   Formulas for [Singularmem](https://github.com/bromso/singularmem).

   ## Usage

   ```bash
   brew install bromso/tap/singularmem
   ```

   ## Available formulas

   - `singularmem` — the CLI binary + MCP server (`singularmem-mcp`)

   Formulas are auto-generated by [cargo-dist](https://github.com/axodotdev/cargo-dist) on every release of the main repo. Do not edit directly.
   ```

8. Create

### B) Mint the `HOMEBREW_TAP_TOKEN` PAT

Fine-grained PAT (recommended over Classic):

1. https://github.com/settings/personal-access-tokens/new
2. Token name: `singularmem release → homebrew-tap`
3. Resource owner: `bromso`
4. Expiration: 1 year (max for fine-grained PATs — needs renewal annually)
5. Repository access: **Only select repositories** → `bromso/homebrew-tap`
6. Repository permissions: **Contents: Read and write**, **Metadata: Read-only** (auto-included)
7. Generate token → copy

Add as repo secret:

1. https://github.com/bromso/singularmem/settings/secrets/actions
2. New repository secret
3. Name: `HOMEBREW_TAP_TOKEN`
4. Value: paste

### C) Verify `dist init` runs cleanly locally before pushing

```
$ dist init

> CI provider: github
> Installers: shell, powershell, homebrew
> Targets: <all 4>
> Generate GitHub release: yes
> Homebrew tap repo: bromso/homebrew-tap
> Tap formula path: Formula/singularmem.rb

  Cargo.toml (workspace metadata)
  .github/workflows/release.yml
```

Both go in the PR.

### D) Order-of-operations for the first release

1. PR for sub-project 7 merges
2. Workspace version bumped, package.json bumped, `v0.15.0` tag pushed
3. Two workflows trigger in parallel:
   - `npm-publish.yml` — builds + (would-be) publishes JS SDK packages (still hits the sub-project 6 CI publish 404 — manual fallback applies)
   - `release.yml` — builds + creates GH Release + pushes Homebrew formula
4. `release.yml` finishes ~5-7 min after the tag
5. GitHub Release `v0.15.0` appears with all 4 platform archives + installers + checksums + Homebrew formula
6. `bromso/homebrew-tap` repo gets a new commit adding `Formula/singularmem.rb`
7. `brew install bromso/tap/singularmem` immediately works

### E) Recovery procedures

`release.yml` has 5 jobs in a DAG. Failures and recovery:

- **`host` fails** → GH Release not created. Fix + re-run the failed job
- **`homebrew-tap` fails** (after `host` succeeded) → GH Release IS published; tap formula missing. Fix the tap auth + re-run. Consumers using curl-installer or manual download unaffected.
- **`build-local-artifacts` per-platform fails** → no release published (build-global depends on all 4). Fix the platform's compile issue + re-tag (or use `dist host --steps=upload --tag=v0.15.0` from local to retry just the upload).

All recovery paths documented in the existing `crates/singularmem-node/RELEASING.md` — sub-project 7 extends that doc with a "CLI binary release" section.

**Decisions locked in this section:**
- Self-hosted Homebrew tap repo (`bromso/homebrew-tap`) created once, owned by the same GitHub user as the main repo. Apache-2.0.
- `HOMEBREW_TAP_TOKEN` fine-grained PAT with `Contents: write` scoped to ONLY the tap repo. 1-year expiration; renewal noted in `RELEASING.md`.
- `dist init` is run by the implementer (interactively) on their local machine. The generated files are part of the PR.
- First release `v0.15.0` ships both JS SDK (via existing `npm-publish.yml` flow) AND binaries (via new `release.yml`) from the same tag.

- Root `Cargo.toml` has a `[workspace.metadata.dist]` block with the Section 2 fields
- `.github/workflows/release.yml` exists and was generated by `dist init` (NOT hand-rolled)
- `dist plan` runs cleanly from the repo root (dry-run validation)

- 4 per-platform archives generated and uploaded to the GitHub Release:

- SHA256 checksum file uploaded alongside

**Installer outputs (per release):**
- `singularmem-installer.sh` — curl-bash installer script, uploaded to GH Release
- `singularmem-installer.ps1` — PowerShell installer script, uploaded to GH Release
- `latest` redirect works for both:
  - `https://github.com/bromso/singularmem/releases/latest/download/singularmem-installer.sh`
  - `https://github.com/bromso/singularmem/releases/latest/download/singularmem-installer.ps1`

**Homebrew tap:**
- `bromso/homebrew-tap` repository exists (one-time setup)
- `HOMEBREW_TAP_TOKEN` secret exists in the main repo
- After each tagged release, `bromso/homebrew-tap` has an updated `Formula/singularmem.rb`
- `brew install bromso/tap/singularmem` works end-to-end on macOS and Linux

**CI workflow behavior:**
- `release.yml` triggers on `v*.*.*` tag pushes

- `release.yml` runs in parallel with the existing `npm-publish.yml` on tag pushes (no race conditions)

**Documentation:**
- `crates/singularmem-node/RELEASING.md` extended with a new "CLI binary release" section

### Constitution Check (v0.2.0)

- **I. Local-first, file-based** ✅ — packaging only; no runtime behavior change.
- **II. Provider-agnostic** ✅ — all four adapters compiled into the shipped CLI.
- **III. Append-only, revisable** ✅ — no data layer changes.
- **IV. Open core** ✅ — Apache-2.0 throughout. Both binaries AND the Homebrew formula are Apache-2.0. The tap repo is Apache-2.0.
- **V. Stable on-disk format** ✅ — `formatVersion` stays `"1"`.
- **VI. Single binary, zero deps** ✅ — **this sub-project delivers on Principle VI for the FIRST TIME.** Until 7 ships, "single binary" was true in theory but consumers had to `cargo install` from source. After 7 ships, `brew install bromso/tap/singularmem` or one-shot curl install gives users a genuinely zero-dependency single binary. The MCP server binary is also a single binary.
- **VII. Composable crates** ✅ — Cargo workspace structure unchanged. The CLI binary statically links the library crates.
- **VIII. Tested at every layer** ✅ — `release.yml`'s `build-local-artifacts` matrix runs on PRs, catching cross-platform compile breakage.
- **IX. Documented behavior** ✅ — RELEASING.md extended; root README extended; tap repo README created.
- **X. Performance budgets** — N/A; packaging.

### Open items (deferred, not blocking)

- **Sub-project 6's CI publish 404 still unresolved.** sub-project 7 doesn't fix it; the JS SDK still ships via manual `npm publish --otp=<TOTP>`. The CLI release via cargo-dist is independent.
- **`HOMEBREW_TAP_TOKEN` expiration in ~1 year.** Fine-grained PATs max out at 1-year expiry. Add a calendar reminder; document the renewal in RELEASING.md.
