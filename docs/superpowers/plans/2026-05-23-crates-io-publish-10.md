## Task 1: Create branch + verify path-dependency version specs

**Files:** none modified yet — read-only audit.

This is a pre-flight check. `cargo publish` refuses to publish a crate that has `path = "..."` dependencies without an accompanying `version = "..."` spec. We need to verify all internal crate dependencies meet this requirement before adding cargo to publish-jobs.

- [ ] **Step 1: Create the sub-project branch**

```bash
cd /Users/jonasbroms/Sites/singularmem
git checkout main
git pull --ff-only
git checkout -b crates-io-publish-10
```

Verify: `git branch --show-current` returns `crates-io-publish-10`.

- [ ] **Step 2: Find all path dependencies in workspace crates**

```bash
grep -rn 'path = ' crates/*/Cargo.toml Cargo.toml | grep -v '#' | head -30
```

Expected: a list of intra-workspace `path = "../singularmem-core"`-style deps. For each, note the surrounding lines.

- [ ] **Step 3: For each path dep, verify it has a `version = "..."` alongside**

A correctly-shaped path dep looks like:

```toml
singularmem-core = { path = "../singularmem-core", version = "0.16.0" }
```

OR a workspace dep that resolves through the root `[workspace.dependencies]`:

```toml
singularmem-core = { workspace = true }
```

The wrong shape (cargo publish refuses):

```toml
singularmem-core = { path = "../singularmem-core" }  # MISSING version
```

For each line returned by Step 2, classify as "ok" (has version OR uses workspace = true) or "needs fix".

If all path deps are ok, skip Step 4. Otherwise, fix in Step 4.

- [ ] **Step 4: (If needed) Fix path deps lacking version specs**

For each crate's Cargo.toml with a bad path dep, edit to add the version. The version should match the workspace version (currently 0.16.0).

Example (use `Edit` tool):

```
OLD: singularmem-core = { path = "../singularmem-core" }
NEW: singularmem-core = { path = "../singularmem-core", version = "0.16.0" }
```

Repeat for each bad dep.

If Step 2 returned only workspace-resolved deps (`{ workspace = true }`), this step is unnecessary.

- [ ] **Step 5: Dry-run `cargo publish` for one crate as a smoke check**

Pick the smallest crate (likely `singularmem-adapter-claude` or similar) and dry-run:

```bash
cd crates/singularmem-adapter-claude
cargo publish --dry-run --allow-dirty 2>&1 | tail -10
cd ../..
```

Expected: completes with `Finished` output. No "missing version" errors. If errors surface, fix and re-dry-run.

The `--allow-dirty` flag is needed because we're on a feature branch with uncommitted future changes.

- [ ] **Step 6: (If Step 4 made changes) Commit**

```bash
git add crates/*/Cargo.toml
git diff --cached --stat
git commit -s -m "chore(crates): add version to internal path-dependencies

cargo publish requires path-dependencies to carry a version = \"...\"
alongside path = \"...\". Adding the version spec lets the crates be
publishable to crates.io. No behavior change — version was already
implicitly that of the workspace.

Sub-project 10."
```

If Step 4 made no changes (all path deps already had versions, or used `workspace = true`), skip the commit. The branch has no commits yet at this point.

cd /Users/jonasbroms/Sites/singularmem

git pull --ff-only

# Order matters: deepest deps first.
for crate_dir in crates/singularmem-core \

                 crates/singularmem-adapter-claude \
                 crates/singularmem-adapter-gemini \

## Risks & rollback

- **Maintainer setup (Task 7) skipped before tagging v0.17.0.** Detection: publish-cargo job fails immediately with auth error or "crate not found" (for namespace not yet claimed). Recovery: complete Task 7 (token + bootstrap), then re-run the failed job or re-tag.
- **Partial publish failure on first CI run.** cargo publish is idempotent on already-published versions (errors with "already exists" which cargo-dist treats as success). Re-run the failed job; or fix the failing crate and re-tag a patch version.
- **`singularmem-node` accidentally published.** Prevented by `publish = false` (cargo refuses). If somehow it slips through, `cargo yank singularmem-node@<version>` immediately; investigate publish-job config.
