# TypeScript SDK Foundation + Reads (5a) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a new `singularmem-node` workspace crate that exposes the read-side of `singularmem-core` (open, get, list, revisions, formatVersion, export) to Node.js / TypeScript via a napi-rs native addon, with a `singularmem` npm package, full type bridge, structured error model, three-layer test strategy, and CI integration.

**Architecture:** A new cdylib crate at `crates/singularmem-node/` uses the napi-rs procedural macros (`#[napi]`) to produce a `.node` binary plus auto-generated `index.js` and `index.d.ts`. The crate is also an npm package (`@napi-rs/cli` drives the build). Async methods run blocking SQLite calls via `tokio::task::spawn_blocking` to keep the JS event loop unblocked. Rust `singularmem_core::Error` variants map to JS `Error` instances with a structured `.code` field for type-safe error handling. Three test layers (Rust unit, Node integration via `node --test`, TS compilation via `tsc --noEmit`) all run in a single new blocking CI job.

**Tech Stack:** napi-rs (napi 2 + napi-derive 2 + napi-build 2 + @napi-rs/cli 2.18), Node 20+ (`node --test`), TypeScript 5.4 (type-check only — no build step for consumers), tokio with `rt-multi-thread` (already in workspace), serde-json (napi feature).

impl From<NodeError> for NapiError {
    fn from(e: NodeError) -> Self {
        let (code, message) = match e.0 {
            CoreError::Validation { field, reason } => (
                "Validation",
                format!("validation failed for {field}: {reason}"),
            ),
            CoreError::SupersedesNotFound { id } => (
                "SupersedesNotFound",
                format!("supersedes target {id} not found"),
            ),
            CoreError::NotFound { id } => (
                "NotFound",
                format!("item {id} not found"),
            ),
            CoreError::AmbiguousLatest { candidates } => (
                "AmbiguousLatest",
                format!("ambiguous latest revision: {} candidates", candidates.len()),
            ),
            CoreError::UnsupportedFormatVersion { found, max_supported } => (
                "UnsupportedFormatVersion",
                format!(
                    "store format version {found} is newer than supported maximum {max_supported}"
                ),
            ),
            CoreError::ReadOnly { operation } => (
                "ReadOnly",
                format!("store is read-only; {operation} requires write access"),
            ),
            CoreError::InvalidId(e) => (
                "InvalidId",
                format!("invalid ULID: {e}"),
            ),
            CoreError::Sqlite { context, source } => (
                "Sqlite",
                format!("SQLite error during {context}: {source}"),
            ),
            CoreError::Io(e) => ("Io", format!("I/O error: {e}")),
            CoreError::Json { context, source } => (
                "Json",
                format!("JSON error during {context}: {source}"),
            ),
        };
        NapiError::new(Status::GenericFailure, message).with_reason(code)
    }
}

/// Build a napi error for invalid store paths surfaced by the binding layer
/// itself (not by the core).
pub fn invalid_store_path(path: &str) -> NapiError {
    NapiError::new(
        Status::GenericFailure,
        format!("store path is not valid: {path}"),
    )
    .with_reason("InvalidStorePath")
}
```

Note: `napi::Error::with_reason` sets the `.code` property on the JS side. If napi 2's actual API names this differently in the installed version, adjust to the correct method (e.g., `set_reason`) but keep the resulting JS shape: a thrown `Error` whose `.code` is the variant string.

- [ ] **Step 3: Fix the `Item` definition and implement `From`**

Replace `src/types.rs` with the working implementation. napi-rs maps Rust `i64` to JS `bigint`, but for an ms-since-epoch we want a JS `Date`. napi-rs maps `chrono::DateTime<Utc>` to JS `Date` natively when the `chrono_date` feature is enabled — alternatively, use `napi::bindgen_prelude::DateTime`. To avoid adding a chrono dep, we keep the Rust-side as `f64` (ms since epoch) annotated to be returned as a JS `Date` via napi's `Date` type:

```rust
//! TypeScript-facing object types and their conversions from core types.

use napi::bindgen_prelude::*;

/// An item retrieved from the store.
///
/// `createdAt` is converted from the core's `jiff::Timestamp` (nanosecond
/// precision) to a JS `Date` (millisecond precision). Sub-millisecond
/// precision is lost.
#[napi(object)]
pub struct Item {
    pub id: String,
    pub content: String,
    /// ms since unix epoch (lossy vs core's ns precision). napi returns this
    /// as a JS `Date` via the `bigint`→`Date` conversion in the binding.
    pub created_at: f64,
    pub supersedes: Option<String>,
    pub tags: Vec<String>,
    pub source: Option<String>,
    pub metadata: serde_json::Value,
}

impl Item {
    #[cfg(test)]
    pub fn created_at_ms(&self) -> i64 {
        self.created_at as i64
    }
}

impl From<singularmem_core::Item> for Item {
    fn from(core: singularmem_core::Item) -> Self {
        Self {
            id: core.id.to_string(),
            content: core.content,
            created_at: core.created_at.as_millisecond() as f64,
            supersedes: core.supersedes.map(|id| id.to_string()),
            tags: core.tags,
            source: core.source,
            metadata: core.metadata,
        }
    }
}

//! The napi-exposed `Store` class — async wrappers around
//! `singularmem_core::Store` that run blocking SQLite work on the tokio
//! blocking pool.

use std::path::PathBuf;
use std::sync::Arc;

use napi::bindgen_prelude::*;
use singularmem_core::{Store as CoreStore, StoreOptions as CoreStoreOptions};

use crate::error::{invalid_store_path, NodeError};

/// Options for `Store.open`.
#[napi(object)]
pub struct StoreOptions {
    /// If true, opens SQLite in read-only mode. Writes will error with
    /// `code: "ReadOnly"`.
    pub read_only: Option<bool>,
}

/// A handle to a Singularmem store on disk.
#[napi]
pub struct Store {
    inner: Arc<CoreStore>,
}

#[napi]
impl Store {
    /// Open a store at the given filesystem path.
    ///
    /// @param path Absolute or relative path to a SQLite file (will be
    ///   created if it does not exist).
    /// @param options Optional `{ readOnly?: boolean }`.
    /// @returns A `Store` instance.
    /// @throws `Error` with `.code` from the standard set (`InvalidStorePath`,
    ///   `Io`, `Sqlite`, `UnsupportedFormatVersion`, …).
    #[napi(factory)]
    pub async fn open(path: String, options: Option<StoreOptions>) -> Result<Self> {
        if path.is_empty() {
            return Err(invalid_store_path(&path));
        }
        let read_only = options.and_then(|o| o.read_only).unwrap_or(false);
        let store_path = PathBuf::from(path);
        let store = tokio::task::spawn_blocking(move || {
            CoreStore::open_with_options(&store_path, CoreStoreOptions { read_only })
        })
        .await
        .map_err(|e| Error::from_reason(format!("tokio join error: {e}")))?
        .map_err(|e| NapiError::from(NodeError::from(e)))?;
        Ok(Self { inner: Arc::new(store) })
    }
}
```

/// List items in the store, ordered oldest → newest by ingest time.
///
/// @param options Optional `{ tags?: string[]; limit?: number }`.
///   When `tags` is given, items must contain ALL listed tags.
///   When `limit` is given, the returned array is capped at that length.
/// @returns Array of items.
#[napi]
pub async fn list(&self, options: Option<ListOptions>) -> Result<Vec<crate::types::Item>> {
    let tags: Vec<String> = options
        .as_ref()
        .and_then(|o| o.tags.clone())
        .unwrap_or_default();
    let limit = options.and_then(|o| o.limit).map(|n| n as usize);
    let inner = self.inner.clone();

    let items = tokio::task::spawn_blocking(move || -> singularmem_core::Result<Vec<singularmem_core::Item>> {
        let iter = if tags.is_empty() {
            inner.list()?
        } else {
            let refs: Vec<&str> = tags.iter().map(String::as_str).collect();
            inner.list_by_tags(&refs)?
        };
        let mut out = Vec::new();
        for item in iter {
            out.push(item?);
            if let Some(n) = limit {
                if out.len() >= n {
                    break;
                }
            }
        }
        Ok(out)
    })
    .await
    .map_err(|e| Error::from_reason(format!("tokio join error: {e}")))?
    .map_err(|e| NapiError::from(NodeError::from(e)))?;

    Ok(items.into_iter().map(Into::into).collect())
}
```

# singularmem (Node.js bindings)

Native Node.js bindings for [Singularmem](https://github.com/bromso/singularmem), a local-first persistent memory layer for LLM workflows.

## Installation

```bash
npm install singularmem
```

> Note: 5a ships source-only builds. Prebuilt platform binaries land in sub-project 6.
> Until then, building from source requires a Rust toolchain.

## Usage

```javascript
import { Store } from 'singularmem';

const store = await Store.open('./memory.db');

const items = await store.list({ tags: ['recipes'], limit: 10 });
for (const item of items) {
  console.log(item.id, item.createdAt, item.content);
}

const oldest = items[0];
const chain = await store.revisions(oldest.id);
console.log(`item has ${chain.length} revisions`);
```

## Read-only mode

```javascript
const store = await Store.open('./memory.db', { readOnly: true });
```

Read-only stores reject every write at the SQLite layer.

## Error handling

All thrown errors have a structured `code` field:

```javascript
try {
  await store.get('not-a-ulid');
} catch (err) {
  if (err.code === 'InvalidId') {
    // ...
  }
}
```

Possible codes: `NotFound`, `Validation`, `SupersedesNotFound`, `AmbiguousLatest`,
`UnsupportedFormatVersion`, `ReadOnly`, `InvalidId`, `InvalidStorePath`,
`Sqlite`, `Io`, `Json`.

## API

See `index.d.ts` for the full TypeScript surface. Currently exposed:

- `Store.open(path, options?)` — async static factory
- `store.get(id)`
- `store.list(options?)` — `{ tags?, limit? }`
- `store.revisions(id)`
- `store.formatVersion()`
- `store.export()` — full JSONL dump

Search, retrieve, and write methods land in subsequent sub-projects.

## License

Apache-2.0
